RULE TYPE 1: MISSING FILE  (Category: Scheduled)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Description:
  Raises an exception when an expected file from a specific partner or system
  does not arrive by its scheduled deadline.

Rule config fields:
  - partner_system_id       (required)  The partner/system expected to send the file
  - frequency               (required)  Cron-style schedule or named preset
      Preset values:
        every_hour     → every hour, every day
        every_10_min   → every 10 minutes, every day
        8am_weekday    → 08:00 AM, every weekday
        5pm_friday     → 05:00 PM, only on Friday
        last_day       → last day of the month at a configurable time
  - file_name_pattern       (optional)  Glob/wildcard string, e.g. "*.txt", "report_*.csv"

Trigger logic:
  - A scheduler runs at the defined frequency (or at end of the cron window)
  - Query: did a transfer arrive from partner_system_id (matching file_name_pattern if set)
    within the current expected window?
  - If NO → create an exception alert
  - Window calculation: the scheduler must determine the expected arrival window
    correctly for each preset (e.g. for every_hour, check the last 60 minutes)

[MISSING-BE-1] Scheduler job for Missing File rules
  - On rule creation, register a recurring job (cron) using the rule's frequency
  - Job execution: query transfers for partner/system within the window, check file name if pattern set
  - If file not found → call exception alert creation service
  - On rule delete/disable → cancel the scheduled job
  - Handle timezone correctly for named presets (org timezone setting)

[MISSING-BE-2] Frequency preset → cron mapping
  - Maintain a mapping of preset values to cron expressions
  - Support for "last day of month" requires special cron handling
  - Document how window edges are handled (e.g. if a file arrives 1 second late)

[MISSING-BE-3] Validate Missing File rule config on POST/PATCH
  - Required: partner_system_id, frequency
  - Optional: file_name_pattern (validate glob syntax if provided)
  - Frequency must be one of the allowed preset values (or valid cron if custom crons are supported)

[MISSING-FE-1] Missing File rule form (already prototyped)
  - Partner or System dropdown (required) → populated from API
  - Expected Frequency dropdown (required) → preset options
  - File Name Pattern text input (optional)
  - Serialize as: { rule_type: "missing", partner_system_id, frequency, file_name_pattern }

[MISSING-FE-2] Auto-generated rule name for Missing File
  - Pattern: "[Partner] was supposed to send a file [named <pattern>] by [frequency label]"
  - Update live as user fills in fields (shown as preview or used as default name)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Description:
  Raises an exception when the number of failed transfers to/from a partner
  or system exceeds a configurable threshold within a rolling time window.

Rule config fields:
  - partner_system_id       (required)  Partner/system to monitor
  - failure_count           (required)  Integer ≥ 1 — number of failures to trigger
  - duration                (required)  Rolling time window
      Allowed values: 5m, 15m, 30m
  - direction               (optional, future) incoming | outgoing | both (default both)

Trigger logic:
  - On every transfer failure event for partner_system_id:
    query count of failures for that partner in the last <duration>
  - If count >= failure_count → create exception alert (deduplicated per window)
  - Consider cooldown: do not re-fire the alert until the window resets or alert is resolved

[FREQ-FAIL-BE-1] Event hook for transfer failures
  - Subscribe to transfer failure events (from transfer processing pipeline)
  - For each failure event: look up all active Frequent Transfer Failures rules
    matching the partner_system_id
  - Evaluate threshold within rolling window using a counter (Redis or DB query)

[FREQ-FAIL-BE-2] Rolling window counter
  - Use a sliding window approach (Redis sorted set keyed by rule_id+partner)
  - Increment on each failure, expire entries older than the window duration
  - Trigger alert when count reaches the threshold

[FREQ-FAIL-BE-3] Alert deduplication
  - Once an alert is triggered for a rule+partner+window, suppress additional alerts
    until the existing alert is resolved or a new window begins

[FREQ-FAIL-BE-4] Validate Frequent Transfer Failures rule config
  - Required: partner_system_id, failure_count (int ≥ 1), duration (enum: 5m|15m|30m)

[FREQ-FAIL-FE-1] Frequent Transfer Failures rule form (already prototyped)
  - Partner or System dropdown (required)
  - Failed Transfers number input (required, min 1)
  - Duration/Period dropdown (required): 5 minutes, 15 minutes, 30 minutes
  - Serialize as: { rule_type: "frequent_failures", partner_system_id, failure_count, duration }

  retried at least a configurable number of times. Optionally scoped by file name.

Rule config fields:
  - partner_system_id       (required)  Partner/system to monitor
  - retry_count             (required)  Integer ≥ 1 — minimum number of retries to trigger
  - file_name_pattern       (optional)  Glob/wildcard string

Trigger logic:
  - On each retry event for a transfer:
    check the current retry count for that transfer against retry_count threshold
  - If retry count >= threshold and partner matches (and file name matches if pattern set):
    → create exception alert
  - One alert per transfer (do not fire on every subsequent retry after threshold is reached)

[RETRIED-BE-1] Event hook for transfer retries
  - Subscribe to transfer retry events (or derive from transfer status change events)
  - For each retry: look up active Transfer Retried rules for the partner

  - On match → create alert, linked to transfer_id; mark transfer as "alerted" to prevent duplicates

[RETRIED-BE-2] Retry count tracking
  - Ensure the transfer record tracks current retry_attempt count
  - The event payload must include this count for efficient evaluation

[RETRIED-BE-3] Validate Transfer Retried rule config

  - Optional: file_name_pattern

Description:
  Raises an exception when a file has been staged (ready for pickup) but
  has not been retrieved by the partner or system within a configurable duration.

Rule config fields:
  - partner_system_id       (required)  Partner/system expected to pick up the file
  - staged_duration         (required)  How long a file may sit staged before alerting
      Allowed values: 30m, 1h, 2h, 4h, 8h
  - file_name_pattern       (optional)  Glob/wildcard string

Trigger logic:
  - When a file is staged for partner_system_id: record the staged_at timestamp
  - A background job (or delay queue) checks after staged_duration:
    has the file been picked up yet?
  - If NOT picked up → create exception alert
  - If file_name_pattern is set, only trigger for matching file names

[STAGED-BE-1] Staged file tracking
  - Ensure the transfer/file record captures:

[STAGED-BE-2] Delayed evaluation job for staged files
  - On "file staged" event: for each active Staged but Not Picked Up rule matching
    the partner, schedule a delayed check for staged_duration in the future
  - At check time: query the file's picked_up_at — if still null → create alert
  - If file_name_pattern set, evaluate before scheduling the check

[STAGED-BE-3] Staged duration values → delay mapping
  - 30m → 30 minutes

  - 4h  → 240 minutes

[STAGED-BE-4] Validate Staged but Not Picked Up rule config

Description:
  Raises an exception (notification) when a file matching a specific name
  pattern is received from a partner or system. Useful for alerting on
  arrival of files that require immediate attention (e.g. invoices, reports).
  Unlike Missing File, this rule fires ON receipt — not on absence.

Rule config fields:
  - partner_system_id       (required)  Partner/system to monitor for incoming transfers
  - file_name_pattern       (required)  Glob/wildcard string to match against received file name
                                        e.g. "invoice_*.xml", "EOD_report_*.csv"

Trigger logic:
  - On transfer receipt event from partner_system_id:
    evaluate file_name_pattern against the received file name
  - If match → create exception alert, linked to transfer_id
  - One alert per matched file/transfer

[FILE-MATCH-BE-1] Event hook for expected file received
  - Subscribe to transfer receipt/completion events
  - For each receipt from a partner: look up active Expected File Received rules
    for that partner_system_id
  - Evaluate glob match of file_name_pattern against the file name
  - On match → create alert

[FILE-MATCH-BE-2] Glob pattern matching utility
  - Implement or reuse a server-side glob/wildcard matching function
    (consistent with any client-side validation of the pattern field)
  - Support: * (any chars), ? (single char), character sets (optional)

[FILE-MATCH-BE-3] Validate Expected File Received rule config
  - Required: partner_system_id, file_name_pattern
  - Validate that file_name_pattern is a non-empty string
  - Optionally validate glob syntax and reject obviously invalid patterns

[FILE-MATCH-FE-1] Expected File Received rule form (already prototyped)
  - Partner or System dropdown (required)
  - File Name Pattern text input (required) — placeholder: "e.g. invoice_*.xml"
  - Serialize as: { rule_type: "file_match", partner_system_id, file_name_pattern }

COMMON RULE FIELDS (applied to all rule types)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Every rule shares these configuration fields regardless of type:

  Field           Type        Notes
  ─────────────── ─────────── ────────────────────────────────────────────
  enabled         boolean     Default: true. Inactive rules are not evaluated.
  severity        enum        critical | high | medium | low
                              Propagated to generated exception alerts.
  assigned_to     user_id     User responsible for reviewing alerts from this rule.
  watchers        user_id[]   Additional users notified when an alert fires.
  custom_name     string      Optional override; if blank, a name is auto-generated.
  description     string      Optional free-text context for the rule.

[COMMON-BE-1] Validate common fields on all rule POST/PATCH
  - severity must be one of: critical, high, medium, low
  - assigned_to must be a valid user_id in the org
  - watchers must all be valid user_ids

[COMMON-BE-2] Auto-generate rule name on the server
  - If custom_name is blank, generate a human-readable name based on rule_type + config
  - Missing File:                "[Partner] was supposed to send [file pattern] by [frequency]"
  - Frequent Transfer Failures: "[Partner] — [N] failures within [duration]"
  - Single Transfer Failure:    "[Partner] — single transfer failure[, file: pattern]"
  - Transfer Retried:           "[Partner] — retried [N]+ times[, file: pattern]"
  - Zero Byte File:             "[Partner] — zero byte file received[, file: pattern]"
  - Staged but Not Picked Up:   "[Partner] — file staged for >[duration][, file: pattern]"
  - Expected File Received:     "[Partner] — file matching [pattern] received"

[COMMON-BE-3] Alert notification on exception trigger
  - When an alert is created: notify assigned_to and watchers
  - Notification channels: in-app, email (configurable per user)
  - Notification payload: rule name, partner, severity, link to exception detail

[COMMON-FE-1] Common fields section in Create Rule modal (already prototyped)
  - Assigned To dropdown (users from API)
  - Severity dropdown: Critical, High, Medium, Low
  - Watchers multi-select or comma-separated text input
  - Custom Name text field (optional)
  - Description textarea (optional)
  - Enabled/Disabled toggle (default on)
